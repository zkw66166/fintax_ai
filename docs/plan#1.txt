 实施步骤

 Phase 1: 数据库基础（无LLM依赖）

 Step 1: config/settings.py
 - DB_PATH = database/fintax_ai.db
 - LLM_API_BASE / LLM_API_KEY / LLM_MODEL 通过环境变量配置（默认 DeepSeek）
 - MAX_ROWS=1000, MAX_PERIOD_MONTHS=36

 Step 2: database/init_db.py — 全部DDL，按依赖顺序创建：
 - 字典表：dim_industry, dim_tax_authority, dim_region
 - 主维表：taxpayer_info + 快照表 taxpayer_profile_snapshot_month, taxpayer_credit_grade_year
 - 明细表：vat_return_general(41指标), vat_return_small(25指标) — DDL原样照搬文档
 - ETL支撑：vat_general_column_mapping, vat_small_column_mapping
 - NL2SQL支撑：vat_synonyms, metric_registry, metric_definition, metric_synonyms
 - 日志表：user_query_log, unmatched_phrases, etl_error_log
 - 视图：vw_vat_return_general, vw_vat_return_small（原样照搬文档）
 - 其他域桩视图（空结构）：vw_EIT_return_annual, vw_EIT_return_quarter, vw_financial_statement, vw_account_balance,
 vw_invoice, vw_enterprise_profile
 - 全部使用 IF NOT EXISTS，幂等可重复执行

 Step 3: database/seed_data.py — 结构性种子数据：
 - 41条一般纳税人栏次映射 + 25条小规模栏次映射（原样照搬文档）
 - ~400条同义词（原样照搬文档），补充 scope_view/taxpayer_type 字段：
   - 一般纳税人同义词 → scope_view='vw_vat_return_general'
   - 小规模同义词 → scope_view='vw_vat_return_small'
   - 共用短语（城建税等）→ scope_view=NULL
 - 字典表示例数据（5个行业、3个税务机关、3个区划）
 - metric_registry/definition/synonyms 示例

 Phase 2: 管线模块（可独立测试）

 Step 5: modules/schema_catalog.py
 - DOMAIN_VIEWS 字典：域→允许视图列表
 - VIEW_COLUMNS 字典：视图→允许列列表（维度+指标完整列举）
 - DENIED_KEYWORDS, DENIED_FUNCTIONS, SYSTEM_TABLES 黑名单

 Step 6: modules/entity_preprocessor.py
 - detect_entities(): 正则提取纳税人ID/名称、期次（"2025年1月"/"2025-01"等）、taxpayer_type
 - normalize_query(): 最长匹配优先 + 不重叠替换同义词，返回 (normalized_query, hits, unmatched)
 - 查 taxpayer_info 确定纳税人类型 → 选择 scope_view → 过滤同义词

 Step 7: modules/sql_auditor.py — 纯正则实现，10条核心规则：
 1. 单语句（无中间分号）
 2. 只读（SELECT/WITH开头）
 3. 禁用危险关键字
 4. 视图白名单（FROM/JOIN对象必须在allowed_views内）
 5. 禁止系统表
 6. 禁止 SELECT *
 7. 必须含 taxpayer_id 过滤
 8. 必须含期间过滤
 9. 必须有 LIMIT
 10. 禁用危险函数

 Phase 3: LLM集成

 Step 8: 提示词文件
 - prompts/stage1_system.txt: 意图解析器，输出严格JSON Schema（domain/views/metrics/filters/clarification）
 - prompts/stage2_vat.txt: VAT域SQL生成器，基于文档6.2.5.1适配

 Step 9: modules/intent_parser.py
 - 使用 openai 库调用 DeepSeek（OpenAI兼容接口）
 - 设置 response_format={"type":"json_object"}
 - 构造用户消息：原始问题 + 已识别实体 + 同义词命中

 Step 10: modules/constraint_injector.py
 - 从阶段1 JSON读取 domain/views/metrics
 - 查 schema_catalog 获取允许列
 - 生成格式化 schema 文本块注入阶段2

 Step 11: modules/sql_writer.py
 - 加载域对应的 system prompt + 动态注入 schema 白名单
 - 调用 LLM 生成 SQL

 Phase 4: 编排与测试

 Step 12: mvp_pipeline.py — 完整管线：
 用户问题 → 实体预处理 → 同义词标准化 → 阶段1(LLM→JSON)
   → 需要澄清? → 返回澄清问题
   → 约束注入 → 阶段2(LLM→SQL) → SQL审核
   → 审核失败? → 重试一次（带错误反馈）
   → 参数化执行 → 记录日志 → 返回结果
 支持命令行和交互模式。

 关键设计决策

 1. LLM: 使用 openai 库 + DeepSeek API（OpenAI兼容），通过环境变量配置
 2. 其他域: 桩视图（空结构），确保管线域路由不崩溃，但不含实际数据
 3. SQL审核: MVP用正则，后续可升级到 sqlglot AST
 4. 同义词scope: 一般/小规模同义词分别标记 scope_view，共用短语留NULL
 5. 修订版本: 默认取最新（ROW_NUMBER + revision_no DESC），测试数据含 rev=0 和 rev=1
