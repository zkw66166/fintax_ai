# 修复计划：三个管线问题

## Context

用户报告了三个 NL2SQL 管线问题：
1. "华兴科技有限公司 加计扣除有哪些政策" 等含公司名+政策关键词的查询错误路由到 financial_data
2. "公司2025年每个季度销售额" 返回空结果
3. "23-25年利润总额、应纳税所得额、所得税税负率季度变化" 三重错误

---

## Bug 1：含公司名的政策查询错误路由

**根因**：`intent_router.py` Layer 0 匹配到纳税人名称"华兴科技有限公司"后直接返回 `financial_data`，不再检查 Layer 1 的税收优惠关键词。当用户查询同时包含公司名和政策/优惠关键词时，Layer 0 的公司名匹配优先级过高。

**日志证据**：
```
[0] 意图路由(stream): financial_data
华兴科技有限公司 加计扣除有哪些政策 → domain_hint=cross_domain → LLM澄清
```

**修复** (`modules/intent_router.py`)：

在 Layer 0 匹配到纳税人名称后，增加"政策意图"检查：如果查询同时包含优惠类关键词（`incentive_keywords` + `extra_incentive`），则不走 `financial_data`，让查询继续到 Layer 1 判断。

具体改动（line 52 附近）：
```python
if matched:
    # 检查是否同时含优惠/政策关键词 → 不拦截，让Layer 1处理
    all_incentive = incentive_kws + ["加计扣除", "即征即退"]
    if not has_exclude and any(k in question for k in all_incentive):
        pass  # 不返回financial_data，继续到Layer 1
    else:
        return "financial_data"
```

注意：需要将 `incentive_kws` 和 `exclude_kws` 的加载提前到 Layer 0 之前（当前在 Layer 1 才加载）。

## Bug 2：单概念时序查询返回空

**根因**："销售额"是 `CONCEPT_REGISTRY` 中"销售金额"的别名，但概念管线要求 `len(concepts) >= 2`（`mvp_pipeline.py:314`），单概念不触发。查询回退到标准 NL2SQL 路径，LLM 对发票域的季度聚合 SQL 生成不稳定。

**修复** (`mvp_pipeline.py`)：

在 Step 3b（指标管线）之后、Step 3c（跨域）之前，新增 Step 3b2：当检测到 ≥1 个概念 + 时间粒度时，无论域是否为 `cross_domain`，都走概念管线生成确定性 SQL。

```python
# Step 3b2: 概念时序查询（单概念+时间粒度即可触发）
concepts = resolve_concepts(resolved_query, entities)
time_gran = entities.get('time_granularity') or detect_time_granularity(resolved_query)
if len(concepts) >= 1 and time_gran and domain != 'cross_domain':
    # 走概念管线
    concept_result = _run_concept_pipeline(conn, entities, concepts, time_gran, resolved_query)
    if concept_result.get('success'):
        # 返回结果
```

同时将 cross_domain 分支中的阈值也从 `>= 2` 改为 `>= 1`（line 314）。

## Bug 3：跨域概念查询三重错误

### 3a："所得税税负率"概念未被概念管线识别

**根因**："所得税税负率"是"企业所得税税负率"的同义词，已在 `financial_metrics_synonyms` 表中注册，属于 `vw_financial_metrics` 视图中的预计算指标（`metric_name='企业所得税税负率'`）。但 `CONCEPT_REGISTRY` 中没有对应条目，`resolve_concepts()` 无法匹配。

**修复** (`modules/concept_registry.py`)：

在 `CONCEPT_REGISTRY` 中新增"企业所得税税负率"概念，直接查询 `vw_financial_metrics`：

```python
'企业所得税税负率': {
    'domain': 'financial_metrics', 'view': 'vw_financial_metrics',
    'column': 'metric_value', 'agg': None,
    'filter': {'metric_name': '企业所得税税负率'},  # 额外过滤条件
    'quarterly_strategy': 'quarter_end',
    'label': '企业所得税税负率',
    'aliases': ['所得税税负率', '所得税税负'],
},
```

同时需要在 `build_concept_sql` 中支持 `filter` 字段，为 `vw_financial_metrics` 域添加 `metric_name` 过滤条件。

`_build_quarterly_sql` 中 `financial_metrics` 域的处理：该视图有 `period_month` 列，按季末月取值即可（与 balance_sheet 类似的 `quarter_end` 策略）。

### 3b："23-25年"未解析为"2023-2025年"

**根因**：`_resolve_relative_dates()` 和年份提取正则均要求 4 位数字年份。"23-25年"格式（2位数字）无匹配规则。

**修复** (`modules/entity_preprocessor.py`)：

在 `_resolve_relative_dates()` 的简单年份替换之前（约 line 469），添加 2 位年份范围正则：

```python
# "23-25年" / "23至25年" / "23到25年" → "2023年到2025年"
def _replace_short_year_range(m):
    y1, y2 = int(m.group(1)), int(m.group(2))
    if y1 < 100: y1 += 2000
    if y2 < 100: y2 += 2000
    return f'{y1}年到{y2}年'

query = re.sub(r'(?<!\d)(\d{2})\s*[-到至]\s*(\d{2})\s*年', _replace_short_year_range, query)
query = re.sub(r'(?<!\d)(\d{2})\s*年\s*[-到至]\s*(\d{2})\s*年', _replace_short_year_range, query)
```

`(?<!\d)` 负向后瞻避免匹配 4 位年份的后 2 位。

### 3c："应纳税所得额"季度查询返回空

**根因**：概念定义使用 `vw_eit_annual_main`（年度视图，无 `period_quarter` 列）。`_build_quarterly_sql` 对 EIT 域生成 `SELECT period_quarter ... FROM vw_eit_annual_main`，该列不存在，返回 0 行。

**修复** (`modules/concept_registry.py`)：

1. EIT 概念定义增加 `quarterly_view` 和 `quarterly_column` 字段：
   - `应纳税所得额`：季度视图用 `vw_eit_quarter_main`，列用 `actual_profit`（季度表中语义最接近的字段）
   - `应纳所得税额`：季度视图用 `vw_eit_quarter_main`，列用 `tax_payable`
2. `_build_quarterly_sql` 增加 `concept_def` 参数，EIT 域优先使用 `quarterly_view`/`quarterly_column`
3. `build_concept_sql` 将 `concept_def` 传递给 `_build_quarterly_sql`

### 3d：多年查询支持

**根因**：`build_concept_sql` 只处理单个 `period_year`，不支持 `period_years` 列表。

**修复** (`modules/concept_registry.py`)：

1. `build_concept_sql` 中：当 `entities['period_years']` 存在时，生成 `period_year IN (:year_0, :year_1, ...)` 条件
2. 所有时间粒度 SQL 构建函数在多年模式下 SELECT 中包含 `period_year`
3. `_period_key_from_row` 支持复合键 `(year, quarter)` / `(year, month)`
4. `_period_label` 支持复合键格式化：`"2023Q1"`, `"2024年3月"`
