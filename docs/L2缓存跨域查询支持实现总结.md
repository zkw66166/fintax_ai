## 背景

### 问题
在2026-03-08之前，L2缓存（SQL模板缓存）完全跳过跨域查询，导致跨域查询无法享受缓存加速。每次跨域查询都需要：
1. LLM生成多个子域SQL（耗时）
2. 执行多个子域SQL（耗时）
3. 合并结果（耗时）

### 用户需求
1. **跨域查询也要支持L2缓存**：每个跨域查询应保存多个子域SQL模板
2. **逐步积累4个模板组合**：
   - 企业会计准则 + 一般纳税人
   - 企业会计准则 + 小规模纳税人
   - 小企业会计准则 + 一般纳税人
   - 小企业会计准则 + 小规模纳税人
3. **包含完整测试方案**：验证不同组合能否准确生成L2模板

---

## 技术方案

### 核心思路
为每个子域SQL单独模板化，保存在一个跨域模板文件中。

### 缓存键设计
```python
# 跨域查询使用复合缓存键（与单域查询相同的域感知策略）
cache_key = MD5(query | response_mode | "cross_domain" | taxpayer_type | accounting_standard)
```

### 存储结构

## 实现细节

### Phase 1: 新增跨域模板化函数

**文件**: `api/services/template_cache.py`

**新增函数**:
1. `templatize_cross_domain_sql(sub_results, company_id)` → (模板化的子域列表, 是否全部成功)
   - 遍历所有子域结果
   - 调用 `templatize_sql()` 模板化每个子域SQL
   - 返回模板化后的子域列表

2. `instantiate_cross_domain_sql(sub_templates, company_id)` → 实例化后的子域SQL列表
   - 遍历所有子域模板
   - 调用 `instantiate_sql()` 实例化每个子域SQL
   - 返回实例化后的子域SQL列表

**修改函数**:
- `save_template_cache()`: 添加 `sub_templates` 和 `cross_domain_operation` 参数
  - 如果提供 `sub_templates`，保存为跨域模板
  - 否则保存为单域模板（现有逻辑）

```python
if l2_cached:
```

## 风险评估

### 中等风险
- 跨域查询逻辑较复杂，需要正确处理多个子域SQL
- 模板实例化后需要重新执行SQL并合并结果
- 需要确保与现有单域L2缓存逻辑兼容

### 缓解措施
- 保持单域L2缓存逻辑不变（向后兼容）
- 跨域逻辑作为新分支添加
- 详细的日志输出便于调试
- 完整的测试场景覆盖所有组合
