# 实施计划：前端流式输出

## Context

fintax_ai 系统有三条查询路由：financial_data（NL2SQL表格）、tax_incentive（本地DB搜索+LLM摘要）、regulation（Coze RAG API）。当前所有路由都是等全部处理完毕后一次性返回结果，用户在等待期间（regulation 5-10s、tax_incentive 2-3s）看不到任何输出。需要改为流式输出，让文本逐步显示。

## 设计方案：Generator 链式传递

采用 Python generator `yield` 模式，从底层模块到 Gradio 前端逐层传递文本片段：

```
regulation_api.query_regulation_stream()  ──yield chunks──┐
                                                          ├─→ mvp_pipeline.run_pipeline_stream() ──yield events──→ app.py query() ──yield──→ Gradio
tax_incentive_query.search_stream()       ──yield chunks──┘
```

- Gradio 4.44.1 原生支持 generator 函数作为事件处理器，每次 `yield` 更新前端组件
- financial_data 路由最终输出是表格（不可流式），保持现有 `run_pipeline()` 不变，仅用进度条
- 现有 `run_pipeline()` 和非流式函数全部保留，零回归风险

## 一、`modules/regulation_api.py` — 新增流式 generator

新增 `query_regulation_stream(question)` generator，与现有 `query_regulation()` 逻辑相同，但每收到一个 SSE `type=answer` 片段立即 yield：

```python
def query_regulation_stream(question, progress_callback=None):
    """Generator: yields (chunk_text, is_done, result_dict)
    - chunk_text: 文本片段（is_done=False 时）
    - is_done: True 表示最后一次 yield
    - result_dict: 完整结果（仅 is_done=True 时有效）
    """
```

- SSE 解析逻辑复用现有代码（encoding fix、card_type 过滤等）
- 每个 `answer_parts.append(content)` 处改为 `yield content, False, None`
- 结束时 `yield "", True, result`
- 异常处理同现有函数，catch 后 `yield "", True, result`
- 现有 `query_regulation()` 保留不动

## 二、`modules/tax_incentive_query.py` — 新增流式方法

### `_summarize_with_llm_stream()`

```python
def _summarize_with_llm_stream(self, question, results, query_intent):
    """Generator: yields (chunk_text, is_done)"""
```

- 与 `_summarize_with_llm()` 相同的 prompt 构建
- DeepSeek API 调用加 `stream=True`，迭代 `chunk.choices[0].delta.content`
- 每个 delta 立即 yield
- 最后 yield 完整累积文本 + `is_done=True`
- LLM 异常时 yield fallback 文本

### `search_stream()`

```python
def search_stream(self, question, limit=20):
    """Generator: yields (chunk_text, is_done, result_dict)"""
```

- 四级搜索部分（结构化→实体→关键词→FTS5）与 `search()` 完全相同（同步执行，很快）
- 搜索完成后，如果有结果，调用 `_summarize_with_llm_stream()` 流式输出摘要
- 无结果时直接 yield 空结果提示
- 现有 `search()` 和 `_summarize_with_llm()` 保留不动

## 三、`mvp_pipeline.py` — 新增 `run_pipeline_stream()`

```python
def run_pipeline_stream(user_query, db_path=None, progress_callback=None):
    """Generator: yields event dicts

    event types:
    - {'type': 'stage', 'route': str, 'text': str}  — 阶段提示
    - {'type': 'chunk', 'text': str}                 — 文本片段
    - {'type': 'done', 'result': dict}               — 最终结果
    """
```

逻辑：
1. 执行意图路由（与 `run_pipeline()` 相同）
2. `route == 'tax_incentive'`：yield stage → 迭代 `tiq.search_stream()` → yield chunks → yield done
3. `route == 'regulation'`：yield stage → 迭代 `query_regulation_stream()` → yield chunks → yield done
4. `route == 'financial_data'`：直接调用现有 `run_pipeline(user_query, progress_callback=progress_callback)` → yield done

financial_data 路由会重复执行一次路由检测（router 是纯正则，<1ms），可忽略。

## 四、`app.py` — `query()` 改为 generator

```python
def query(user_input, history, progress=gr.Progress()):
    """Generator: 逐步 yield 6 个输出到 Gradio 组件"""
    if not user_input.strip():
        yield "请输入查询问题", "", "", "", history, _history_choices(history)
        return

    NO = gr.update()  # 不更新的组件占位符

    def progress_callback(pct, desc):
        progress(pct, desc=desc)

    accumulated = []
    route = None

    for event in run_pipeline_stream(user_input, progress_callback=progress_callback):
        if event['type'] == 'stage':
            route = event.get('route')
            label = '📚 税收优惠政策' if route == 'tax_incentive' else '🌐 法规知识库'
            yield f"{label}\n\n⏳ {event['text']}", "", "", "", NO, NO

        elif event['type'] == 'chunk':
            accumulated.append(event['text'])
            partial = "".join(accumulated)
            label = '📚 税收优惠政策' if route == 'tax_incentive' else '🌐 法规知识库'
            yield f"{label}\n\n{partial}▌", NO, NO, NO, NO, NO

        elif event['type'] == 'done':
            result = event['result']
            # ... 与现有 query() 相同的结果格式化 + 历史记录逻辑 ...
            yield main_output, entity_text, intent_text, sql_text, new_history, dropdown_update
```

关键设计：
- `gr.update()` 保持组件当前值不变，避免闪烁
- `▌` 光标字符在流式输出时显示打字效果，最终 yield 时移除
- 首次 yield 清空调试面板（entity/intent/sql 设为 `""`）
- 历史记录仅在最终 yield 时更新
- financial_data 路由不经过 chunk 循环，直接到 done 事件，行为与当前完全一致
