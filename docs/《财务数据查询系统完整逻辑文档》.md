## 1. 系统简介

### 系统定位

fintax_ai 是一个中文税务与财务咨询平台，将自然语言查询转换为针对 SQLite 结构化税务申报和财务数据的 SQL 查询。系统采用 DeepSeek LLM 驱动的两阶段 NL2SQL 管线，针对不同查询类型优化了多条执行路径。

### 支持的9大领域

1. **增值税申报（VAT）** - 增值税纳税申报表
2. **企业所得税（EIT）** - 企业所得税（年报 & 季报）
3. **资产负债表（Balance Sheet）** - 财务状况报表
4. **科目余额（Account Balance）** - 总账科目余额表
5. **利润表（Profit Statement）** - 损益表
6. **现金流量表（Cash Flow）** - 现金流量表
7. **财务指标（Financial Metrics）** - 计算型财务比率
8. **发票（Invoice）** - 进项发票和销项发票
9. **企业画像（Enterprise Profile）** - 公司画像聚合

### 核心特性

- **多轮对话** - 跨对话轮次的实体继承
- **跨路由综合** - 混合分析（财务数据 + 税收优惠 + 法规知识）
- **确定性路径** - 概念注册表和计算指标绕过LLM，保证精度
- **两级持久化缓存** - L1（完整结果）+ L2（SQL模板）+ 智能适配
- **双会计准则支持** - 企业会计准则（EAS）vs 小企业会计准则（SAS）
- **双纳税人类型** - 一般纳税人 vs 小规模纳税人

## 2. 架构总览

### 系统流程

```
用户查询（自然语言）
    ↓
API入口 (/api/chat)
    ↓
公司名称解析
    ↓
意图路由器（Layer 0）
    ├─→ financial_data → NL2SQL管线（6条执行路径）
    ├─→ tax_incentive → 本地数据库搜索 + LLM摘要
    └─→ regulation → 外部Coze RAG API
    ↓
SSE流式响应
    ↓
显示格式化
    ↓
前端（React SPA）
```

### 核心组件

| 组件 | 用途 | 技术栈 |
|------|------|--------|
| FastAPI后端 | REST API + SSE流式 | Python 3.9+ |
| SQLite数据库 | 结构化财务数据存储 | SQLite 3 |
| DeepSeek LLM | 意图解析 + SQL生成 | deepseek-chat (V3) |
| 意图路由器 | 路由分类 | 关键词 + 模糊匹配 |
| NL2SQL管线 | 查询 → SQL → 结果 | 多阶段LLM管线 |
| 概念注册表 | 确定性跨域查询 | 240+预注册概念 |
| 指标计算器 | 计算型财务比率 | 15+公式 |
| 对话管理器 | 多轮上下文 | 实体继承 |
| 缓存系统 | L1 + L2持久化 | 文件存储，LRU淘汰 |

### 技术栈

- **后端**: FastAPI, SQLite, OpenAI SDK（DeepSeek兼容）
- **前端**: React, Chart.js, SSE（Server-Sent Events）
- **LLM**: DeepSeek-V3（deepseek-chat模型）
- **认证**: JWT（HS256，24小时过期）
- **缓存**: 文件存储持久化缓存，MD5键

## 3. 查询入口

### API端点

**POST /api/chat** - SSE流式端点

### 请求格式

```json
{
  "query": "华兴科技2025年1月增值税",
  "company_id": "91110000MA01234567",
  "response_mode": "detailed",
  "thinking_mode": "quick",
  "conversation_history": [],
  "conversation_depth": 3,
  "conversation_enabled": true
}
```

**参数说明**:

- `query` (string, 必填) - 自然语言查询
- `company_id` (string, 可选) - 纳税人识别号，用于公司范围查询
- `response_mode` (string) - "detailed" | "concise"（默认："detailed"）
- `thinking_mode` (string) - "quick" | "think" | "deep"（默认："quick"）
- `conversation_history` (array) - 多轮对话的历史记录
- `conversation_depth` (int) - 考虑的对话轮数（2-5，默认：3）
- `conversation_enabled` (bool) - 启用多轮对话功能

### 响应格式（SSE）

端点流式传输三种事件类型：

**1. stage事件** - 路由指示器
```
event: stage
data: {"route": "financial_data", "text": "正在查询财务数据..."}
```

**2. chunk事件** - 文本片段（用于tax_incentive/regulation路由）
```
event: chunk
data: {"text": "根据查询结果..."}
```

**3. done事件** - 最终结果
```
event: done
data: {
  "success": true,
  "route": "financial_data",
  "domain": "vat",
  "sql": "SELECT ...",
  "results": [...],
  "display_data": {...},
  "interpretation": "...",
  "cache_hit": false,
  "cache_source": "l1"
}
```

### 公司名称解析

当提供 `company_id` 时，系统自动解析公司名称并前置到查询中：

```python
# 原始查询: "2025年1月增值税"
# 公司ID: "91110000MA01234567"
# 解析名称: "华兴科技"
# 最终查询: "华兴科技 2025年1月增值税"
```

这确保实体预处理器能正确提取纳税人信息。

### 原始查询保留

对于 `tax_incentive` 和 `regulation` 路由，系统传递 `original_query`（不含公司名称前缀），避免公司名称片段污染关键词搜索：

```python
# 用户查询: "资产税收优惠有哪些"
# 公司前缀查询: "华兴科技 资产税收优惠有哪些"
# 原始查询（用于tax_incentive）: "资产税收优惠有哪些"
```

## 4. 意图路由器

### 三大主路由

意图路由器在领域检测**之前**将用户查询分类为三条主路由：

1. **financial_data** - 针对结构化财务数据库的查询
2. **tax_incentive** - 税收优惠政策搜索（本地数据库）
3. **regulation** - 程序性/法规知识（外部RAG API）

### 多层分类系统

路由器使用**5层关键词分类**系统（Layer -2 到 Layer 1 + 默认）：

#### Layer -2: 财务数据优先
**触发条件**: 数据类关键词 + 税种/财务关键词同时出现
```python
# 数据关键词: "数据", "查询", "多少", "金额", "税额"
# 税种关键词: "增值税", "所得税", "资产负债表", "利润表"
# 示例: "查询华兴科技增值税数据" → financial_data
```

#### Layer -1: 知识库优先
**触发条件**: 程序性/操作性关键词
```python
# 关键词: "如何", "怎么", "流程", "步骤", "办理", "申报流程"
# 示例: "如何办理增值税申报" → regulation
```

#### Layer 0: 企业数据查询
**触发条件**: 纳税人名称匹配（精确或模糊前缀）+ 时间/金额模式
```python
# 纳税人名称: ["华兴科技", "鑫源贸易", ...]
# 模糊前缀: "华兴" 匹配 "华兴科技"（长度 ≥ 2）
# 时间模式: r'\d{4}年.*多少'
# 示例: "华兴2025年增值税" → financial_data
```

**特殊处理**: 如果纳税人名称匹配但查询同时包含优惠/法规关键词，不拦截 - 让Layer 1处理。

#### Layer 1: 税收优惠
**触发条件**: 优惠关键词（排除黑名单）
```python
# 优惠关键词: "优惠", "减免", "免税", "退税", "补贴"
# 额外关键词: "加计扣除", "研发费用加计扣除"
# 排除关键词: "优惠政策申报", "优惠备案流程"（程序性）
# 示例: "资产税收优惠有哪些" → tax_incentive
```

#### 默认: 法规知识
如果没有层匹配，默认路由到 `regulation`。

### 配置文件

路由器行为由 `config/tax_query_config.json` 控制，支持热重载：

```json
{
  "financial_db_priority_keywords": ["数据", "查询", "多少"],
  "financial_tax_type_keywords": ["增值税", "所得税"],
  "knowledge_base_priority_keywords": ["如何", "怎么", "流程"],
  "incentive_keywords": ["优惠", "减免", "免税"],
  "exclude_from_incentive": ["申报", "备案", "流程"],
  "extra_incentive_keywords": ["加计扣除"],
  "regulation_passthrough_keywords": ["流程", "步骤", "办理"]
}
```

### 主开关

```python
# config/settings.py
ROUTER_ENABLED = True  # 设为False绕过路由，恢复原始行为
```

## 5. 领域检测

### 两阶段检测

领域检测分为**两个阶段**：

1. **启发式检测**（entity_preprocessor.py）- 快速基于关键词的分类
2. **LLM确认**（intent_parser.py）- Stage 1 LLM验证并细化领域

### 9大领域优先级顺序

检测遵循以下优先级（从高到低）：

```
1. 财务指标（独特关键词）
2. 现金流量表（独特关键词）
3. 科目余额（时间性 + 方向性关键词）
4. 利润表（时间性关键词或基于月份的默认）
5. 资产负债表（时间性关键词或项目名称）
6. 企业所得税（时间性关键词）
7. 发票（显式"发票"关键词）
8. 增值税（税务查询的默认）
9. 跨域升级（检测到多个领域关键词）
```

### 领域检测规则

#### 1. 财务指标
**关键词**: "财务指标", "毛利率", "ROE", "净资产收益率", "资产负债率", "流动比率"
```python
# 示例: "华兴科技2025年ROE" → financial_metrics
```

#### 2. 现金流量表
**关键词**: "现金流量表", "经营活动现金流", "投资活动现金流", "筹资活动现金流"
```python
# 示例: "2025年经营活动现金流" → cash_flow
```

#### 3. 科目余额
**高优先级关键词**（科目余额独有）:
- "科目余额", "借方发生额", "贷方发生额", "期初余额", "期末余额"
- "总账", "明细账", "科目汇总表"

**中优先级关键词**（需要上下文）:
- "发生额", "借方", "贷方", "会计科目"

**科目特定名称**（不在资产负债表主项目中）:
- "银行存款", "库存现金", "主营业务收入", "本年利润"
- "坏账准备", "累计折旧", "原材料", "库存商品"

**消歧逻辑**:
```python
# 如果查询包含高优先级关键词 → account_balance
# 如果查询包含中优先级关键词 + 科目特定名称 → account_balance
# 如果查询包含"期初"/"期末"但也有资产负债表项目 → balance_sheet
```

#### 4. 利润表
**时间性关键词**: "本期金额", "本年累计", "本月", "本季度"
**项目关键词**: "营业收入", "营业成本", "净利润", "利润总额"

**与企业所得税消歧**:
- "年度"/"季度" → EIT（年度/季度税务申报）
- "本期金额"/"本年累计" → Profit（会计报表）
- 共享项目（净利润、利润总额）→ 默认Profit，除非出现"年度"/"季度"

#### 5. 资产负债表
**时间性关键词**: "年初", "期初", "年末", "期末"
**项目关键词**: "资产", "负债", "所有者权益", "流动资产", "固定资产"

**与科目余额消歧**:
- "年初"/"期初" + 资产负债表主项目 → balance_sheet
- "期初"/"期末" + 科目特定名称 → account_balance
- "借"/"贷"/"发生额" → account_balance

#### 6. 企业所得税（EIT）
**时间性关键词**: "年度", "季度", "年报", "季报"
**项目关键词**: "应纳税所得额", "应纳所得税额", "减免所得税额"

**视图选择**:
- "年度"/"年报" → `vw_eit_annual_main`
- "季度"/"季报" → `vw_eit_quarter_main`

#### 7. 发票
**显式关键词**: "发票"（必须存在）
**方向关键词**:
- "进项发票", "采购发票" → `vw_inv_spec_purchase`
- "销项发票", "销售发票" → `vw_inv_spec_sales`

**优先级**: 在增值税之前检查，避免误判（增值税查询常提及"发票"）

#### 8. 增值税（VAT）
**关键词**: "增值税", "VAT", "销项税", "进项税", "应纳税额"
**默认**: 如果没有其他领域匹配且查询与税务相关

**视图选择**:
- 一般纳税人 → `vw_vat_return_general`
- 小规模纳税人 → `vw_vat_return_small`

#### 9. 跨域升级
如果检测到主领域但查询包含其他领域的关键词，升级为跨域：

```python
# 示例: "华兴科技2025年1月增值税和利润对比"
# 主领域: VAT
# 次要关键词: "利润"（profit领域）
# 结果: 升级为跨域查询
```

### 范围感知视图选择

系统根据以下条件自动选择正确的视图：
1. **领域**（vat, eit, balance_sheet, profit, cash_flow, account_balance）
2. **纳税人类型**（一般纳税人 vs 小规模纳税人）
3. **会计准则**（企业会计准则 vs 小企业会计准则）

```python
# 示例: 资产负债表
# 一般纳税人 + 企业会计准则 → vw_balance_sheet_eas
# 小规模纳税人 + 小企业会计准则 → vw_balance_sheet_sas

# 示例: 增值税
# 一般纳税人 → vw_vat_return_general
# 小规模纳税人 → vw_vat_return_small
```

**视图映射规则**:
- **财务报表**（balance_sheet, profit, cash_flow, account_balance）: 仅按 `accounting_standard` 区分，与 `taxpayer_type` 无关
- **增值税申报**: 仅按 `taxpayer_type` 区分
- **企业所得税申报**: 不按纳税人类型或会计准则区分；仅区分年报 vs 季报

## 6. 执行路径选择

### 决策树

系统根据查询特征选择**6条执行路径**之一：

```
查询到达financial_data路由
    ↓
启用多轮对话？→ 否 → 继续路径选择
    ↓ 是
检查对话历史路由
    ↓
≥2种不同路由？→ 否 → 继续路径选择
    ↓ 是
LLM检测到综合需求？→ 否 → 继续路径选择
    ↓ 是
路径6: 混合分析（Tier 2）✓
    ↓
[如果不是路径6，继续...]
    ↓
启用多轮对话 + 历史仅包含financial_data？
    ↓ 是
路径5: 多轮对话（Tier 1）✓
    ↓ 否
[继续确定性路径检查...]
    ↓
检测到计算指标？（Step 2b）
    ↓ 是
路径4: 计算指标✓
    ↓ 否
检测到≥2个概念 + 时间粒度？
    ↓ 是
路径3: 概念注册表✓
    ↓ 否
检测到跨域操作？
    ↓ 是
路径2: 跨域查询✓
    ↓ 否
路径1: 标准单域查询✓
```

### 路径优先级顺序

1. **路径6: 混合分析**（最高优先级，首先检查）
2. **路径5: 多轮对话**（Tier 1）
3. **路径4: 计算指标**（确定性）
4. **路径3: 概念注册表**（确定性）
5. **路径2: 跨域查询**（基于LLM）
6. **路径1: 标准查询**（默认，基于LLM）

### 触发条件汇总

| 路径 | 触发条件 | 使用LLM？ |
|------|----------|-----------|
| 路径6 | 多轮启用 + ≥2路由 + 需要综合 | 是（税务筹划专家） |
| 路径5 | 多轮启用 + 历史仅financial_data | 是（Stage 1+2） |
| 路径4 | 指标关键词匹配（如"ROE"、"资产负债率"） | 否（确定性） |
| 路径3 | ≥2概念 + 时间粒度 + 非多期间 | 否（确定性） |
| 路径2 | 检测到≥2个领域 | 是（每个子域Stage 1+2） |
| 路径1 | 单领域 | 是（Stage 1+2） |

## 7. 路径1：标准单域查询

### 流程（7个步骤）

```
步骤1: 实体预处理
    ↓
步骤2: Stage 1 LLM（意图解析器）
    ↓
步骤3: 约束注入
    ↓
步骤4: Stage 2 LLM（SQL编写器）
    ↓
步骤5: SQL审计
    ↓
步骤6: SQL执行
    ↓
步骤7: 结果格式化
```

### 步骤1: 实体预处理

**目的**: 提取实体并标准化查询

**操作**:
- 相对日期解析（"今年3月" → "2026年3月"）
- 纳税人ID/名称提取（正则 + 模糊匹配）
- 期间提取（年、月、季度）
- 纳税人类型检测
- 领域检测（启发式）
- 同义词标准化（最长匹配优先）

**输出**:
```python
{
  "taxpayer_id": "91110000MA01234567",
  "taxpayer_name": "华兴科技",
  "taxpayer_type": "一般纳税人",
  "period_year": 2025,
  "period_month": 1,
  "domain_hint": "vat",
  "normalized_query": "华兴科技 2025年1月 增值税申报表"
}
```

### 步骤2: Stage 1 LLM（意图解析器）

**目的**: 将用户意图解析为结构化JSON

## 9. 路径3：概念注册表（确定性）

### 240+预定义概念

**概念类型**:
- 直接取值（agg=None）- 报表项目，直接SELECT
- 聚合取值（agg=SUM）- 明细项目，GROUP BY聚合

## 13. 缓存策略（L1 + L2）

### 两级持久化缓存架构

fintax_ai 采用两级持久化缓存系统，L1和L2均为文件存储，服务器重启后数据保留，且具备公司感知能力。

### 缓存流程图

```
请求到达
    ↓
thinking_mode == "quick" or "think"?
    ↓ 是
L1缓存检查 (Line 49-76)
    ↓ 命中
返回完整缓存结果（跳过所有pipeline）
    ↓ 未命中
L2缓存检查 (Line 79-144)
    ↓ 命中
实例化SQL → 执行 → 返回（跳过Stage 1+2）
    ↓ 未命中
L2智能适配 (Line 148-223)
    ↓ 命中
适配SQL → 执行 → 保存新L2 → 返回
    ↓ 未命中
执行完整pipeline (Line 233-325)
    ├─→ Stage 1: Intent Parser (LLM)
    ├─→ Stage 2: SQL Writer (LLM)
    ├─→ SQL Audit
    ├─→ SQL Execution
    └─→ Display Formatting
    ↓
保存到L1 (Line 286-294)
保存到L2 (Line 297-314)
    ↓
返回结果
```

### L1缓存：完整结果缓存

#### 作用

存储完整的查询结果，包括：
- SQL执行结果（results）
- 显示数据（display_data）
- 数据解读（interpretation）
- 路由信息（route）
- 所有元数据

#### 缓存键

```
MD5(company_id | normalized_query | response_mode)
```

示例：
- 查询: "华兴科技2025年1月增值税"
- company_id: "91110000MA01234567"
- response_mode: "detailed"
- 缓存键: MD5("91110000MA01234567|华兴科技2025年1月增值税|detailed")

#### 存储结构

文件路径: cache/{cache_key}.json

文件内容包含：
- cache_key: 缓存键
- company_id: 公司ID
- query: 查询文本
- response_mode: 响应模式
- route: 路由类型
- result: 完整结果对象
- interpretation: 数据解读文本
- created_at: 创建时间
- accessed_at: 最后访问时间
- access_count: 访问次数

#### 介入时机

检查时机（最早介入，Line 49-76）：
- Quick模式: 返回缓存结果 + 缓存解读（0次LLM调用）
- Think模式: 返回缓存结果 + 设置need_reinterpret=True（前端触发新解读）

保存时机（pipeline执行完成后，Line 286-294）：
- 剥离瞬态键后保存完整结果

#### LRU淘汰策略

- 最大文件数: 1500
- 淘汰逻辑: 按访问时间排序，删除最旧文件
- 内存索引: 服务器启动时从磁盘重建，快速查找

#### 三种模式交互

| 模式 | L1缓存行为 | LLM调用 | 前端行为 |
|------|-----------|---------|---------|
| Quick | 返回缓存结果 + 缓存解读 | 0次 | 直接显示 |
| Think | 返回缓存结果 + 设置need_reinterpret | 0次（pipeline），1次（解读） | 触发/api/interpret |
| Deep | 完全绕过L1缓存 | 2次（Stage 1+2） | 显示新结果 |

### L2缓存：SQL模板缓存

#### 作用

存储SQL模板（将taxpayer_id替换为占位符），实现：
- 跨公司复用: 同类型查询可用于不同公司
- 智能适配: 财务报表可在会计准则间自动适配（_eas ↔ _sas）

#### 缓存键（域感知，2026-03-06重构）

财务报表（balance_sheet, profit, cash_flow, account_balance）：
```
MD5(query | mode | "fs" | accounting_standard)
```

增值税：
```
MD5(query | mode | "vat" | taxpayer_type)
```

企业所得税：
```
MD5(query | mode | "eit")
```

未知域（向后兼容）：
```
MD5(query | mode | taxpayer_type | accounting_standard)
```

#### 存储结构

文件路径: cache/template_{cache_key}.json

文件内容包含：
- cache_key: 缓存键
- query: 查询文本
- response_mode: 响应模式
- taxpayer_type: 纳税人类型
- accounting_standard: 会计准则
- domain: 领域
- sql_template: SQL模板（带占位符）
- intent: Stage 1意图JSON
- created_at: 创建时间

#### 模板化与实例化

模板化：
- 将SQL中的taxpayer_id替换为占位符:taxpayer_id
- 示例: WHERE taxpayer_id = '91110000MA01234567' → WHERE taxpayer_id = :taxpayer_id

实例化：
- 将占位符替换为实际taxpayer_id
- 示例: WHERE taxpayer_id = :taxpayer_id → WHERE taxpayer_id = '91110000MA01234567'

#### 介入时机

检查时机（L1未命中后，Line 79-144）：
1. 获取纳税人类型和会计准则
2. 检测查询的域类别
3. 查找L2缓存
4. 如果命中，实例化SQL并执行
5. 返回结果（跳过Stage 1+2）

保存时机（pipeline执行完成后，Line 297-314）：
1. 模板化SQL
2. 获取纳税人类型和会计准则
3. 保存模板到L2缓存

#### 智能适配（财务报表专用）

目的: 当L2缓存未命中但存在对立会计准则的缓存时，自动适配SQL

适配逻辑（Line 148-223）：
1. 检测cache_domain为"financial_statement"
2. 查找对立会计准则的缓存
3. 适配SQL（交换_eas ↔ _sas视图后缀）
4. 实例化并执行适配后的SQL
5. 保存新的L2缓存

适配规则：
- 企业会计准则 → 小企业会计准则: _eas → _sas
- 小企业会计准则 → 企业会计准则: _sas → _eas

限制: 增值税查询不适配，因为列结构差异显著：
- 一般纳税人: output_tax, input_tax, tax_payable
- 小规模纳税人: tax_due_total

#### LRU淘汰策略

- 最大文件数: 500
- 淘汰逻辑: 与L1相同，按文件访问时间排序删除最旧文件

## 14. 特殊查询处理

### 单期间查询

示例: "华兴科技2025年1月增值税"
- period_year = 2025, period_month = 1
- SQL: WHERE period_year = 2025 AND period_month = 1

### 多期间查询

月份范围:
- 示例: "华兴科技2025年1-3月增值税"
- period_year = 2025, period_months = [1, 2, 3]
- SQL: WHERE period_year = 2025 AND period_month IN (1, 2, 3)

年度对比:
- 示例: "华兴科技2024年和2025年增值税对比"
- period_years = [2024, 2025]
- SQL: WHERE period_year IN (2024, 2025)

### 季度查询

示例: "华兴科技2025年Q1增值税"
- period_year = 2025, period_quarter = 1
- 扩展: period_months = [1, 2, 3]
- SQL: WHERE period_year = 2025 AND period_month IN (1, 2, 3)

### 会计准则选择

自动选择基于taxpayer_info:
- 一般纳税人 → 企业会计准则（EAS）
- 小规模纳税人 → 小企业会计准则（SAS）

视图映射:
- 资产负债表: EAS → vw_balance_sheet_eas, SAS → vw_balance_sheet_sas
- 利润表: EAS → vw_profit_eas, SAS → vw_profit_sas
- 现金流量表: EAS → vw_cash_flow_eas, SAS → vw_cash_flow_sas

### 纳税人类型处理

增值税视图:
- 一般纳税人 → vw_vat_return_general
- 小规模纳税人 → vw_vat_return_small

列结构差异:
- 一般纳税人: output_tax, input_tax, tax_payable
- 小规模纳税人: tax_due_total

### 相对日期解析

当前年/月:
- "今年3月" → "2026年3月"
- "本月" → 当前月（VAT上下文感知）

前期:
- "去年12月" → "2025年12月"
- "上个月" → 前一个月
- "上个季度" → 前一个季度范围

### 空数据处理

"本年"/"今年"查询返回0行时:
- 检查数据库中的最大period_year
- 如果max_year < current_year:
  - 返回友好消息: "该企业暂无{current_year}年数据，最新数据截至{max_year}年。请尝试指定具体年份查询。"

## 15. SQL生成与审计

### 领域特定Stage 2提示词

9个提示词位于prompts/目录:

2. stage2_eit.txt - 企业所得税申报

4. stage2_profit.txt - 利润表
5. stage2_cash_flow.txt - 现金流量表
6. stage2_account_balance.txt - 科目余额
7. stage2_financial_metrics.txt - 财务指标
8. stage2_invoice.txt - 发票
9. stage2_cross_domain.txt - 跨域查询

### 约束注入

允许的视图: 来自Stage 1 + 模式目录白名单
允许的列: 来自选定视图
最大行数: 1000（可通过MAX_ROWS配置）

### SQL审计规则

硬阻止:
- 非SELECT语句（INSERT, UPDATE, DELETE, DROP等）
- 多语句查询（分号分隔）
- 不允许的视图/列（不在白名单中）
- 缺少taxpayer_id过滤器
- 缺少期间过滤器（领域特定）
- 无LIMIT子句
- SELECT *（必须指定列）
- 危险函数（DROP, DELETE, UPDATE, EXEC等）

领域特定检查:
- EIT: 季报视图需要period_quarter过滤器
- 月度领域: 需要period_month过滤器
- 利润表/现金流量表: time_range验证

表别名前缀模式: 支持period_year*100+period_month模式

### 重试逻辑

审计失败时:
- 带反馈重试一次给LLM
- 反馈包含具体审计错误

执行失败时:
- 带错误消息重试一次给LLM
- 包含SQLite JOIN限制提示


```
    ↓

    ↓
意图路由器
    ├─→ financial_data
    │   ├─→ 路径6: 混合分析
    │   ├─→ 路径5: 多轮对话
    │   ├─→ 路径4: 计算指标
    │   ├─→ 路径3: 概念注册表
    │   ├─→ 路径2: 跨域查询
    │   └─→ 路径1: 标准查询
    ├─→ tax_incentive
    │   └─→ 本地数据库 + LLM摘要
    └─→ regulation
        └─→ Coze RAG API
    ↓
SSE流式响应
    ↓
显示格式化
    ↓
前端
```

### 路径选择决策树

```
查询到达financial_data路由
    ↓
多轮 + ≥2路由 + 综合? → 是 → 路径6
    ↓ 否
多轮 + 仅financial_data? → 是 → 路径5
    ↓ 否
检测到计算指标? → 是 → 路径4
    ↓ 否
≥2概念 + 时间? → 是 → 路径3
    ↓ 否
检测到跨域? → 是 → 路径2
    ↓ 否
路径1（标准）
```

